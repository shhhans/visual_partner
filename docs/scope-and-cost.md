### 1.2 已实现部分的取舍说明

- **U2/U3 的"自动断句 + 打断"是体验地基。** 没有按钮意味着回合边界完全交给 ASR 的
  `sentence_end` 与端侧语音活动判定。barge-in 不只是体验问题，也是成本问题（见 C6）：
  被打断的内容不再继续付费生成。
- **U4/U7 共用一条"按需视觉"主线。** 视觉永远不是常驻视频流，而是两种触发：
  用户主动问（U4，工具触发）、画面自己变了（U7，端侧 diff 触发）。VL 描述统一以文本入
  ReAct 历史，所以"记得刚看到的"是天然副产物——这条主线同时是最大的成本杠杆（见 C1）。
- **U8 的记忆是后台异步的。** 写入（落库 + 事实抽取）走进程级单写者 worker，不占回复热路径；
  召回在回复前用当前用户语句做全文检索，以临时 system 消息回灌。agent 还能用 `remember`
  主动记、`correct_memory` 自愈纠错。细节见 README「长期记忆」。

### 1.3 U9 怎么实现的：纯 prompt，复用主动视觉链路

U9 没有写新代码，全靠调整三处 core prompt 让既有的主动视觉链路承担——
链路本身（端侧 diff → `scene_change` → VL 描述入史 → 空闲时 `_proactive_reply` 决策）早已就位：

1. **`SCENE_VL_PROMPT`（VL 描述）** —— 要求每次都说明"是否有人在场"，并在有人时补充
   穿着、神情、在做什么。这样历史里连续的「[系统检测到画面变化]」观察就天然带上了
   "在场/不在场"和用户状态信息。
2. **`PROACTIVE_DIRECTIVE`（是否开口）** —— 让 LLM 对照历史观察识别"用户从『不在画面』
   变成『重新出现』"的跳变，此时贴心地问一句"刚刚干什么去了"；留意到穿着/神情值得关心
   时也可自然搭话；拿不准就 SKIP。
3. **`SYSTEM_PROMPT`（人设）** —— 立住"像在乎用户的朋友"的底色：留意穿搭、情绪、动向，
   适度关心、不啰嗦。

> 关键在于"在场/不在场"的跳变判断是交给 **LLM 读历史观察**完成的，不是显式状态机。
> 优点是零新增代码、随 prompt 即可调；代价是判定稳定性依赖 VL 描述质量与历史长度。
> 若后续发现误触发（短暂遮挡被当成"离开"），再考虑把"离开区间"显式建模（见第三节开放问题）。

## 二、运营成本控制

核心信条：**云调用按量计费，凡能在端侧免费做的、凡能不调的，就不调。**
四个云依赖里，ASR/TTS 按音频时长或字数计费、LLM/VL 按 token 计费，所以省钱的着力点
分别是"少送音频""少念字""少调 VL""少喂 token"。

### 2.1 想到的技巧 vs 实际采用

> 编号（C1…C16）为本文内部引用用，与源码注释里残留的旧编号（沿用自 stale 的 design-doc）不对应，以本文为准。

| # | 技巧 | 状态 | 说明 |
|---|---|---|---|
| C1 | **视觉按需化**：VL 只由工具/变化触发，闲聊零视觉成本 | ✅ | 对比常驻视频流，成本差一个数量级以上——最大杠杆 |
| C2 | **画面只在显著变化时才调 VL**：端侧 diff 流水线判定 | ✅ | 帧差/EMA/hysteresis/关键帧对比全在端侧，仅"真变了"才上行 |
| C3 | 帧端侧预处理：缩图 ≤768px + JPEG 压缩再上传 | ✅ | VL 按图块计 token，缩图直接降单价 |
| C4 | 模型分层：LLM 用 qwen-plus 当大脑而非 max | 🟡 | LLM 默认 qwen-plus✅；VL 默认仍是 qwen-vl-max，降级 vl-plus 只是 `.env` 可配项，当前未启用 |
| C5 | system prompt 约束短回答、口语化 | ✅ | 输出 token 是大头，且 TTS 按字计费，短回答双重省 |
| C6 | barge-in 即停 LLM 生成与 TTS 合成 | ✅ | 被打断的内容不再付费生成/合成 |
| C7 | VL 结果短期缓存：同帧 N 秒内复用描述 | ✅ | 同帧 8s 复用（`_VL_CACHE_TTL`）；**仅被动路径**（`look_at_camera`），主动视觉刻意不缓存（每次变化要对当前画面重判） |
| C8 | 主动视觉冷却 + SKIP 沉默：闲置/无人时零云成本 | ✅ | 双层防抖：前端 hysteresis + 服务端冷却 8s（`_SCENE_COOLDOWN`）；再由 LLM 判定不值得即 SKIP，不发声不二次调 |
| C9 | 记忆事实抽取走 SKIP 模式：闲聊回合不产生事实 | ✅ | 绝大多数回合抽取返回 SKIP，控住后台 LLM 抽取成本 |
| C10 | 端侧免费能力优先：回声消除/降噪/缩图/diff 都不上云 | ✅ | getUserMedia 约束 + Canvas 像素 diff |
| C11 | **端侧 VAD 减少 ASR 触发**：静默期不送音频 | 🟡 | capsule 已有 Silero VAD，但目前只驱动 UI，**未门控上行**；详见 2.2 |
| C12 | CLIP/嵌入捕捉**语义**变化触发 VL（而非仅像素 diff） | ⬜ | 当前只有像素级 diff；语义变化（同位置换了东西但像素差不大）抓不到 |
| C13 | 部署**本地 TTS**替代云 cosyvoice | ⬜ | TTS 按字计费且是高频调用，本地化能直接归零这部分账单 |
| C14 | 部署**本地 ASR**（如 sherpa/faster-whisper）替代云 paraformer | ⬜ | 与 C11 互补：本地 ASR 后，连续上行也不再产生云费用 |
| C15 | Context cache：命中 system prompt 隐式缓存 | ⬜ | 长 system prompt 时收益明显，待评估供应商支持 |
| C16 | token/计费用量采集，把"省了多少"变成可观测 | ⬜ | 目前只采集延迟，未采集 token；成本优化缺闭环度量 |

### 2.2 C11 实情：VAD 有，但还没省到钱

这是本文最需要说清的一处。**capsule 端确实加载了 Silero VAD（端侧 ONNX，能区分人声
与键盘/敲击噪声），RMS 能量门限作为加载失败时的兜底**——但它当前的产出 `vadSpeaking`
只用于**驱动 UI**（监听/说话状态、胶囊展开动画），并没有去**门控上行音频**：

```
// clients/capsule/src/hooks/useVisualPartner.ts（约 276 行）
mic = await startMic((pcm) => {
  if (ws?.readyState === WebSocket.OPEN) ws.send(pcm);  // 无条件上传，未看 vadSpeaking
}, ...)
```

web 端则连 VAD 都没有，`clients/web/main.js` 同样是 WS 一开就连续上传每一块 PCM。
也就是说：**无论有没有人说话，云 ASR 都在持续接收音频、持续计费。**

所以严格讲，C11 应记为 🟡 部分实现——"减少 ASR 触发"这个**成本收益还没拿到**。
要拿到它，改动其实很小：在 onPcm 回调里按 `vadSpeaking`（及一小段尾窗，避免吞掉句尾）
决定是否 `ws.send(pcm)`，静默期不上行。风险点是别把句首/句尾切掉影响 ASR 断句，
需要配合启停余量调参。**这是一个低成本、高确定性的优化，建议优先做。**

### 2.3 未实现项的优先级判断（一家之言）

- **C11（VAD 门控上行）> C13（本地 TTS）> C14（本地 ASR）> C12（CLIP 语义触发）**。
- C11 改动最小、收益直接、且 VAD 基础已就位，应最先做。
- C13/C14 是把"按量付费的云"换成"一次性部署的本地"，对长期高频使用收益最大，但引入
  部署/资源/质量权衡，属于中期工程。C13 优先于 C14：TTS 调用更频繁、按字计费更直观。
- C12 解决的是"像素没怎么变但语义变了"的漏检（例如桌上同一位置换了本书），属于**召回质量**
  问题而非**成本**问题，且需要额外的端侧嵌入模型，性价比靠后。
- **C16（用量采集）应尽早补**：现在所有成本结论都是定性的，没有 token 度量就无法验证
  上述任何优化"到底省了多少"。它本身不省钱，但让其余每一项变得可衡量。
